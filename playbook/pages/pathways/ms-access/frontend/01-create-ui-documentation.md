# Document Forms

This page provides guidance on documenting forms as part of the MS Access migration pathway. Documenting forms is a critical first step in understanding and modernising legacy applications, ensuring that user needs and business processes are fully captured before technical migration begins.

## Why Document Forms?

- To capture the structure, fields, and interactions of each form in the legacy system
- To ensure all user needs and business requirements are understood
- To provide a foundation for user story generation and GDS-compliant UI design

## How to Document Forms

1. **Identify all forms** in the legacy MS Access application.
2. For each form, record:
   - Form name and purpose
   - All fields, controls, and interactive elements
   - Validation rules and business logic
   - User roles and workflows
3. Use screenshots or detailed descriptions to support your documentation.

## Generating User Stories

Once forms are documented, generate comprehensive user stories for each form. Follow the GDS (gov.uk) service design standards to ensure accessibility, usability, and clarity.

- See the [Access Form User Stories prompt](../../../appendix/prompt-library/ms-access/access-form-user-stories.md) for a detailed template and instructions.
